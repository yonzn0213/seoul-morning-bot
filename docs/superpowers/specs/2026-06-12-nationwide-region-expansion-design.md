# 전국 시/군/구 지역 확장 설계

- 날짜: 2026-06-12
- 상태: 승인됨 (구현 plan 대기)

## 배경

현재 봇은 **서울 25개 구**만 지원한다. `common.py`의 `REGIONS` dict에 구별 기상청 격자좌표(nx, ny)가 하드코딩되어 있고, 미세먼지는 에어코리아 `getCtprvnRltmMesureDnsty`를 `sidoName=서울`로 호출해 "측정소 이름 == 구 이름" 매칭으로 처리한다. 지역 선택은 1단계 인라인 키보드다.

이를 **전국 시/군/구 단위**로 확장한다. 유저는 먼저 시/도를 고르고, 그 안에서 세부 시/군/구를 고른다.

## 목표 / 비목표

**목표**
- 전국 17개 시/도 → 각 시/군/구(약 250개) 2단계 선택
- 시/군/구별 날씨(기상청 단기예보)
- 시/군/구 측정소 매칭, 없으면 시/도 평균으로 미세먼지 제공

**비목표 (YAGNI)**
- 읍/면/동 단위 세분화
- TM좌표 기반 최근접 측정소 조회
- 기존 서울 유저 마이그레이션 (현재 `state.json`의 users가 비어 있어 불필요)

## 핵심 결정

| 항목 | 결정 | 근거 |
|------|------|------|
| 세분화 단위 | 시/군/구 (~250) | 정확도와 버튼 수의 현실적 균형. 읍면동은 과함 |
| 미세먼지 | 시군구 측정소 매칭 + 시/도 평균 fallback | 측정소명이 시군구명과 항상 일치하진 않음. 광역시는 잘 맞고 도는 평균 대체 |
| 격자좌표 확보 | 정적 파일 `regions.json` 번들 | 좌표가 변할 일 없음, 런타임 API 추가 호출 없음, 안정적 |
| 지역 식별 | `시도 + 시군구` 조합 | "중구", "남구" 등 중복 구명 존재 |
| callback_data | 인덱스 인코딩 | 한글 UTF-8 3바이트/자 → 64바이트 제한 회피, payload 최소화 |

## 데이터 모델 — `regions.json` (신규)

```jsonc
{
  "서울특별시": {
    "airkorea": "서울",                       // 에어코리아 sidoName 파라미터
    "sigungu": { "강남구": {"nx": 61, "ny": 126}, "...": {} }
  },
  "경기도": {
    "airkorea": "경기",
    "sigungu": { "수원시": {"nx": 60, "ny": 121}, "...": {} }
  }
  // ... 17개 시/도 전체
}
```

- `nx`/`ny`: 해당 시군구 소속 읍면동들의 격자좌표 **중앙값**(median)을 대표값으로 사용.
- `airkorea`: 에어코리아 `sidoName` 값 (서울, 부산, 대구, 인천, 광주, 대전, 울산, 세종, 경기, 강원, 충북, 충남, 전북, 전남, 경북, 경남, 제주).

### 데이터 생성 (구현 시 1회)

기상청 공식 "단기예보 조회서비스 격자_위경도" 데이터에서 시도/시군구/읍면동 + 격자 X/Y를 읽어 시군구별 median 격자좌표로 집계해 `regions.json`을 생성한다. 생성 스크립트는 `tools/build_regions.py`로 남겨 재현 가능하게 한다.

**검증 (생성 시 필수)**
- 17개 시/도 모두 존재
- 각 시군구 좌표가 기상청 격자 유효 범위 내 (nx 1~149, ny 1~253)
- 시군구 누락/중복 없음 (행정구역 기준 개수와 대조)
- `airkorea` 값이 에어코리아 허용 sidoName 집합에 속함

## 상태 스키마 — `state.json`

```jsonc
{
  "offset": 0,
  "users": {
    "<chat_id>": { "sido": "경기도", "sigungu": "수원시", "name": "..." }
  }
}
```

기존 `region` 단일 필드 → `sido` + `sigungu`로 변경.

## 2단계 키보드 흐름 (`register.py`, `common.py`)

```
/start, /region
   └─> 시/도 키보드 (17개, 3열)         callback_data = "s:<sido_idx>"
          └─ 시/도 탭
              └─> 시군구 키보드 + "⬅ 뒤로"  callback_data = "r:<sido_idx>:<sigungu_idx>"
                                            뒤로 = "s:back"
                     └─ 시군구 탭
                         └─> 저장 + 확정 메시지 (editMessageText)
```

- 인덱스는 `regions.json`의 **삽입 순서**(Python 3.7+ dict 순서 보장)를 기준으로 한다. 키보드 생성과 콜백 파싱이 같은 순서를 공유해야 한다.
- `⬅ 뒤로` 버튼으로 시/도 선택으로 복귀.
- 알 수 없는/범위 밖 인덱스는 안내 메시지로 graceful 처리.

## 브리핑 로직 (`briefing.py`)

- **날씨**: 현행 유지. `(nx, ny)` 단위 `weather_cache` — 같은 격자를 쓰는 시군구는 1회 호출 공유.
- **미세먼지**:
  - 유저들이 속한 **시/도별로** `getCtprvnRltmMesureDnsty`를 호출하고 시/도 단위로 캐싱 (`dust_cache[airkorea]`).
  - `dust_for(sigungu, sido_dust)`: 시군구명과 일치하는 측정소가 있으면 그 값(`is_avg=False`), 없으면 그 시/도 평균(`is_avg=True`).
  - 메시지의 "(서울 평균)" 표기 → "(<시도> 평균)"으로 일반화.

## 모듈 책임

| 파일 | 변경 |
|------|------|
| `regions.json` | 신규. 시/도 → 시군구 → 격자좌표 + airkorea |
| `tools/build_regions.py` | 신규. regions.json 생성·검증 스크립트 (재현용) |
| `common.py` | `REGIONS` 하드코딩 제거 → `load_regions()`. `sido_keyboard()`, `sigungu_keyboard(sido)` 추가. 인덱스 인코딩/디코딩 헬퍼 |
| `register.py` | 2단계 콜백 핸들링 (`s:`, `r:`), `⬅ 뒤로`, 새 상태 스키마 저장 |
| `briefing.py` | 시/도별 dust 캐싱, `dust_for` 시그니처 변경, 평균 표기 일반화, 유저 region 조회를 sido/sigungu로 |
| `README.md` | "서울" → "전국", 2단계 선택 안내, 봇 소개 문구 |

## 테스트

**단위 테스트**
- 인덱스 콜백 라운드트립: `(sido_idx, sigungu_idx)` → callback_data → 다시 시도/시군구명 복원
- `regions.json` 스키마/좌표 범위 검증 (테스트로도 재확인)
- `dust_for`: 측정소 매칭 hit / miss(시도 평균 fallback) 두 경로
- 키보드 행 분할(3열) 로직

**수동 테스트**
- `/start` → 시/도 선택 → `⬅ 뒤로` → 다른 시/도 → 시군구 선택 → 저장 확인
- `register.py` 수동 실행으로 폴링 동작
- `briefing.py` 수동 실행으로 실제 브리핑 수신 (광역시 구 + 도 시군구 각각 1건씩)

## 리스크

- **격자 원본 데이터 확보**가 핵심 단계. 신뢰할 수 있는 공식/검증된 출처에서 받아 좌표 범위·시군구 누락을 반드시 검증한다. 데이터가 부정확하면 엉뚱한 지역 날씨가 나간다.
- 도 단위 미세먼지는 평균이라 체감과 차이날 수 있음 — 메시지에 "(시도 평균)" 명시로 완화.
